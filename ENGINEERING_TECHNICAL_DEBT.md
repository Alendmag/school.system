### P0-01: حساب GPA بأرقام عشوائية وهمية

**الوصف**
في `server/services.ts` (السطر ~45)، عندما لا تتوفر درجات للطالب، يُحسَب GPA بالمعادلة:
```
const baseGPA = 85 + (studentIndex % 15);
```
هذا يُعطي أرقاماً بين 85 و99 لكل طالب بدون درجات — بيانات كاذبة تظهر في الواجهة كحقيقة.

**لماذا هو مشكلة**
أي Dashboard أو Report يعتمد على GPA يعرض معلومات مضللة. إذا بُني منطق أعمال على هذه القيمة (مثل قبول الطلاب، أو تصفية الأوائل) سيعطي نتائج خاطئة.

**الأثر على الأعمال**: عالٍ جداً — قرارات إدارية مبنية على بيانات كاذبة  
**الأثر على المستخدم**: يرى GPA لطلاب ليس لديهم درجات مسجلة  
**الأثر التقني**: يُفسد صحة كل التقارير المالية والأكاديمية  
**تعقيد الإصلاح**: منخفض — إرجاع `null` بدل الرقم الوهمي  
**خطر الانحدار**: منخفض — تغيير سلوك واحد  
**التبعيات**: لا شيء  
**الحل الموصى به**: إرجاع `null` عند غياب الدرجات، والتعامل مع `null` في كل مكان يعرض GPA  
**قبل أم بعد الـ Backend**: **قبل** — يجب إصلاحه حتى مع البيانات الوهمية

**الأثر على الأعمال**: كارثي — مخالفة قانونية محتملة  
**الأثر على المستخدم**: لا حماية للخصوصية  
**الأثر التقني**: لا يمكن نشر النظام إنتاجياً بدونه  
**تعقيد الإصلاح**: عالٍ — يتطلب login screen, sessions, JWT, middleware  
**خطر الانحدار**: عالٍ — سيُغيّر تدفق التطبيق كاملاً  
**التبعيات**: يتطلب Backend أولاً  
**الحل الموصى به**: Supabase Auth (email/password) + RLS policies على كل الجداول  
**قبل أم بعد الـ Backend**: **بعد** البنية التحتية مباشرة

### P1-01: AppContext — سياق ضخم واحد يُعيد تصيير كل شيء

**الوصف**
`AppContext` يحتوي على: اللغة + الثيم + الدور + المؤسسة + قاعدة البيانات الكاملة (500+ طالب، 150 فاتورة، 150 درجة...). أي تغيير في أي خاصية يُعيد تصيير كل مستهلكي الـ context في شجرة المكونات.

**لماذا هو مشكلة**
إضافة طالب واحد → تحديث `db` → إعادة تصيير الـ Sidebar, Header, Dashboard, Finance, Reports... جميعاً في نفس الوقت. مع نمو البيانات سيتراكم هذا الأثر.

**الأثر على الأعمال**: أداء بطيء مع البيانات الحقيقية  
**الأثر على المستخدم**: تأخر ملموس عند الإضافة أو التعديل  
**الأثر التقني**: يُصعّب debugging، ويحول دون استخدام React DevTools بفاعلية  
**تعقيد الإصلاح**: عالٍ — يتطلب تقسيم Context أو الانتقال لـ Zustand/TanStack Query  
**خطر الانحدار**: عالٍ — تغيير معماري يمس كل الصفحات  
**التبعيات**: يُفضَّل بعد إتمام Backend (لأن TanStack Query سيحل المشكلة تلقائياً)  
**الحل الموصى به**: بعد Backend، تُستبدل `db` في context بـ TanStack Query cache. اللغة والثيم يبقيان في context منفصل خفيف  
**قبل أم بعد الـ Backend**: **بعد** — البنية التحتية ستحل هذا ضمنياً

### P1-02: CommandPalette — استخدام window.dispatchEvent + setTimeout

**الوصف**
```typescript
// CommandPalette.tsx:41
window.dispatchEvent(new CustomEvent('open-add-student'));
setTimeout(() => navigate('/students'), 150);
```
لفتح نافذة الإضافة، يُرسَل حدث DOM عالمي ثم يُنتظر 150ms. المستمع في `Students.tsx` يستقبل الحدث ويفتح الـ Sheet.

**لماذا هو مشكلة**
- إذا تغيّر اسم الحدث في مكان دون الآخر — لا خطأ في TypeScript
- الـ 150ms تعسفية — على أجهزة بطيئة قد لا تكفي
- ربط ضمني بين مكونَين لا يعرف أحدهما الآخر
- لا يمكن اختباره بالوحدات (unit testing)

**الأثر على الأعمال**: منخفض حالياً لكن يتراكم مع الوقت  
**الأثر على المستخدم**: احتمال عدم فتح النافذة على أجهزة بطيئة  
**الأثر التقني**: يُصعّب إضافة إجراءات جديدة للـ CommandPalette  
**تعقيد الإصلاح**: متوسط — استبدال بـ Command Registry في Context  
**خطر الانحدار**: منخفض — محدود في مكونَين  
**التبعيات**: لا شيء  
**الحل الموصى به**: `CommandRegistry` في Context يحتفظ بـ Map من اسم الأمر إلى دالة. الصفحات تُسجّل أوامرها عند mount، CommandPalette يستدعيها مباشرة  
**قبل أم بعد الـ Backend**: **قبل** — مستقل عن الخلفية

**الأثر التقني**: ازدواجية مصادر البيانات  
**تعقيد الإصلاح**: منخفض — استبدال array hardcoded بـ `db.notifications`  
**خطر الانحدار**: لا شيء  
**التبعيات**: لا شيء حالياً. بعد Backend يتصل بـ API  
**الحل الموصى به**: استخدام `services.getNotificationsForUser(currentUser.id, currentUser.role)` مباشرة  
**قبل أم بعد الـ Backend**: **قبل** — سهل ويُصلَح الآن

### P1-04: CRUD ناقص — لا تعديل ولا حذف في أي صفحة

**الأثر على المستخدم**: محاصَر مع بيانات خاطئة لا يمكن تصحيحها  
**الأثر التقني**: إكمال CRUD هو شرط أساسي لأي نظام ERP  
**تعقيد الإصلاح**: عالٍ — يتطلب نماذج تعديل لكل كيان + تأكيد الحذف  
**خطر الانحدار**: متوسط — إضافة حالات جديدة  
**التبعيات**: يُفضَّل بعد Backend لأن التعديل يجب أن يُحفَظ  
**الحل الموصى به**: نمط موحد: EditModal<T> يأخذ الكيان الحالي ويحدّث بـ API call  
**قبل أم بعد الـ Backend**: **بعد** — لا قيمة للتعديل بدون persistence

### P1-05: التحقق من TypeScript — npm run check لا يُشغَّل

**الوصف**
لا يوجد CI يُشغّل `tsc --noEmit` تلقائياً. الكود يمكن أن يحتوي على أخطاء TypeScript صامتة لا تُكتشَف.

**لماذا هو مشكلة**
TypeScript بدون تشغيل المُتحقق هو مجرد JavaScript مع تلميحات. الأخطاء تُكتشَف فقط عند crash في runtime.

**الأثر على الأعمال**: منخفض مباشرة  
**الأثر على المستخدم**: crashes مفاجئة  
**الأثر التقني**: يُلغي فائدة TypeScript في الحماية  
**تعقيد الإصلاح**: منخفض — تشغيل `tsc --noEmit` كجزء من البناء  
**خطر الانحدار**: لا شيء  
**التبعيات**: لا شيء  
**الحل الموصى به**: إضافة `tsc --noEmit` كخطوة في build script، وإصلاح أي أخطاء ظاهرة  
**قبل أم بعد الـ Backend**: **قبل** — مستقل وسهل

**التوصية**: **C** — بعد Backend، TanStack Query سيتولى كل state البيانات. يبقى Context للغة والثيم فقط (خفيف جداً). لا داعي لـ Zustand.
